# Openclaw本地git仓库配置手册

> 目标：让任意模型代理在接管现有 OpenClaw 环境时，稳定完成升级，并保证“可追踪、可回退、可汇报”。

---

## 0. 先决条件（未满足则不启动）

在开始前，代理必须拿到并确认四个输入：

1. 本次允许改动的范围（目录、配置、插件、Agent）
2. 升级目标（版本目标或能力目标）
3. 风险偏好（稳妥/平衡/激进）
4. 失败策略（自动回退或人工确认后回退）

输入不完整时，先补齐，再执行。

---

## 1. 处理当前 `~/.openclaw` 内容（接管阶段）

代理接管已有环境时，不得直接“整体重做”或“整仓重排”。

必须先完成内容分层：

### 1.1 资产盘点

识别并标注：

- 核心配置（如主配置、Agent 配置、绑定关系）
- 运行数据（会话、日志、缓存、临时文件）
- 内容资产（文档、草稿、脚本、技能）
- 高风险数据（密钥、凭证、令牌）

### 1.2 分类处置

将现有内容按四类处理：

- **保留**：当前仍在使用、且对稳定性有价值
- **整理**：需要归档、重命名、规范化
- **隔离**：敏感或不应进入版本管理的内容
- **清理候选**：确认无效后可移除的历史残留

### 1.3 原则

- 不在未分层前直接大规模改动
- 不把运行垃圾和敏感信息纳入版本管理
- 所有结构调整都要可解释、可复核

---

## 2. 本地仓库注册与身份初始化（强制询问）

代理在建立或接管本地仓库身份时，必须先询问用户：

1. 使用哪个账户名（Git identity name）
2. 使用哪个邮箱（Git identity email）

### 2.1 默认选项（用户可一键选择）

如果用户不提供，代理必须给出“使用默认值”选项，并自动生成：

- 默认账户名：基于当前 Agent 名称和主机标识生成
- 默认邮箱：使用本地占位邮箱规则生成（避免真实隐私邮箱暴露）

### 2.2 记录要求

无论用户自定义还是默认生成，都要在报告中记录：

- 本次采用的身份策略（用户指定 / 默认生成）
- 是否已完成身份初始化

---

## 3. 变更闭环：每次变更都必须提交（强制）

代理执行升级时，必须遵循：

1. 每个原子变更都要形成独立提交
2. 不允许“改完不提交”或“攒多批再提交”
3. 提交说明必须清楚表达“改动对象 + 改动目的”

### 3.1 提交粒度要求

- 一次提交只覆盖一个明确目的
- 禁止把无关文件混入同一提交
- 关键配置与普通文档应分开提交

---

## 4. 提交后必须向用户报告（强制）

每完成一次提交，代理都要立刻给用户发送变更回报，至少包含：

1. 提交标识（commit hash）
2. 改动范围（涉及目录/文件）
3. 改动目的（一句话）
4. 风险提示（如有）
5. 回退提示（该提交可撤销）

要求：不延迟到任务结束再一次性汇总。

---

## 5. 升级执行主流程（阶段化）

## 阶段 A：基线采集

采集当前版本、配置拓扑、插件状态、权限边界、仓库状态。

## 阶段 B：风险评估

标注高/中/低风险，给出触发条件与缓解方案。

## 阶段 C：变更计划

把升级拆成原子步骤；每步写清目标、对象、成功条件、失败动作。

## 阶段 D：实施与即时提交

每个原子步骤完成后立即提交，并立即向用户回报。

## 阶段 E：验证

覆盖配置层、运行层、业务层，逐项给出通过/失败/降级通过结论。

## 阶段 F：交付

输出完整升级报告，标注最终状态、未决问题、后续建议。
